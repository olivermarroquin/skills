# Profile: `content-expansion-ev`

Ranks content expansion candidates — new section types for existing pages and new page types — generated by the `competitor-architecture-diff` skill. Designed to sequence expansion work alongside the existing Core-30 build.

## Scoring mode

`weighted-sum`

Expansion items trade off against each other on multiple dimensions: a zero-effort schema addition might beat a high-demand new page type because it ships immediately. Weighted-sum allows this trade-off.

## Criteria

### 1. `effort`

- **Label:** Implementation effort
- **Description:** How much work to build this expansion. Lower effort = faster value capture.
- **Direction:** `lower-is-better`
- **Weight:** 0.30
- **Values:** `[low, medium, high]`
- **How to populate:** From the architecture-diff backlog's `effort` field.

### 2. `data_availability`

- **Label:** Data availability
- **Description:** Whether the expansion is supportable from existing data sources or needs new research.
- **Direction:** `higher-is-better`
- **Weight:** 0.25
- **Values:** `[supportable, partially-supportable, needs-new-data]`
- **How to populate:** From the architecture-diff backlog's `our_data_status` field.

### 3. `demand_signal`

- **Label:** Demand signal strength
- **Description:** Evidence of search demand for this expansion — competitor URL count, keyword presence, search intent alignment.
- **Direction:** `higher-is-better`
- **Weight:** 0.25
- **Values:** `[very-high, high, medium-high, medium, low]`
- **How to populate:** From the architecture-diff backlog's `demand_signal` assessment.

### 4. `reuse_potential`

- **Label:** Cross-client reuse potential
- **Description:** Can this expansion type be reused for other clients or is it project-specific?
- **Direction:** `higher-is-better`
- **Weight:** 0.20
- **Values:** `[high, medium, low]`
- **How to populate:** From the architecture-diff backlog's `reuse_potential` field.

## Hard dependency constraints

- `howto-schema` must come before `speakable-schema` (speakable references HowTo content)
- New section types should generally precede new page types (enrich existing pages before creating new ones)
