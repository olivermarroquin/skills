# Handoff Frontmatter Spec

The shape every handoff file follows. This is the data contract for the skill's DECOMPOSE (write) and AUDIT (read + edit) modes.

## Required frontmatter

```yaml
---
type: handoff
status: active
created: YYYY-MM-DD
purpose: One-sentence description of what the chat will produce.
tags: [handoff, <project-slug-or-domain>, <other-applicable-tags>]
---
```

Field-by-field:

- `type`: always `handoff`. No variants.
- `status`: lifecycle state. Three values:
  - `active` — the handoff is queued or in-flight; the work has not landed.
  - `consumed` — the handoff has been used; the work has landed. Add a `consumed:` date.
  - `cancelled` — the handoff is no longer relevant (priorities shifted, approach changed, blocker became permanent). Add a `cancelled:` date and a `cancellation-reason:` line.
- `created`: ISO date the handoff was first written.
- `purpose`: one sentence. Names the deliverable concretely. Avoid "explore" or "investigate" — handoffs produce shipped artifacts.
- `tags`: always includes `handoff` and the project slug (if part of a project) or the primary domain tag (if standalone). Other applicable tags follow.

## Adding `consumed:` when the work lands

When a handoff is consumed, the chat consuming it edits the frontmatter as part of the closing protocol:

```yaml
---
type: handoff
status: consumed
created: YYYY-MM-DD
consumed: YYYY-MM-DD
purpose: <unchanged>
actual-deliverable: <one-line summary of what actually shipped — file paths and outcome>
tags: <unchanged>
---
```

- `consumed:`: ISO date the work landed.
- `actual-deliverable:`: optional but strongly recommended. One line summarizing the shipped artifact paths and outcome. Some chats use a longer "Actual deliverable" blockquote at the top of the body instead — either is acceptable, but at least one of them must exist for the audit to pass.

## Adding `cancelled:` when a handoff is killed

```yaml
---
type: handoff
status: cancelled
created: YYYY-MM-DD
cancelled: YYYY-MM-DD
cancellation-reason: One-line reason (priorities shifted, approach changed, etc.)
purpose: <unchanged>
tags: <unchanged>
---
```

Don't delete cancelled handoffs. They are useful breadcrumbs when reconstructing how a thread evolved or why a previously-planned chat never ran.

## "Actual deliverable" blockquote convention

Many consumed handoffs add a blockquote at the top of the body summarizing what shipped. Example from the consumed `handoff-2026-05-26-perplexity-refinement-skill.md`:

```markdown
> **Consumed 2026-05-27.** Wave 0 of the perplexity-skill-build roadmap.
> Chat session expanded scope mid-flight to also ship Wave 1A (auto-rec
> upgrades), Wave 1B (perplexity-research-suite router + shared references
> folder), and Wave 1C (perplexity-citation-monitoring + query-shapes
> references + first ev-electric citation-history note). Full audit trail
> in the running consolidated execution log:
> [[execution-log-2026-05-27-perplexity-refinement-skill-build|exec log]].
> Outputs on disk:
> - `~/workspace/skills/perplexity-refinement/`
> - `~/workspace/skills/perplexity-research-suite/`
> - `~/workspace/skills/perplexity-citation-monitoring/`
> - `~/workspace/skills/perplexity-shared/`
> - First worked refinement example at <path>
```

The blockquote sits between the H1 heading and the "Prompt to paste into a new chat" section. It is read by AUDIT to confirm the deliverable list is present.

## Body structure (generated by DECOMPOSE)

DECOMPOSE generates handoff bodies following this shape:

```markdown
# Handoff — <human-readable name>

<2-3 sentence framing — what the chat produces, why it matters, prerequisites if any>

## Prompt to paste into a new chat

--- start prompt ---

<full prompt body — context to read, what you're building, success criteria, deliverables, out of scope>

## Status

- Created YYYY-MM-DD <reason>
- Prereqs: <list>
- Spawn-readiness: <Ready / Tier-2 / Tier-3 with explanation>
- Estimated build time: <range>

## Closing protocol (run BEFORE telling the operator the chat is done)

<seven-step closing protocol inserted verbatim from closing-protocol-template.md>

--- end prompt ---

## See also

- [[<parent project README or related handoffs>]]
- [[<related blueprints, primers, or specs>]]
```

The `--- start prompt ---` and `--- end prompt ---` markers are mandatory. The operator's copy-paste workflow uses them to delimit the prompt body — everything between the markers gets pasted into a new Cowork chat verbatim.

The closing protocol sits **inside** the prompt block (between Status and `--- end prompt ---`) so the chat consuming the handoff sees it as part of its instructions.

## File naming

- **Standalone handoff:** `handoff-YYYY-MM-DD-<topic-slug>.md` — date-stamped, kebab-case, max 80 chars.
- **Project-folder handoff:** `phase-N-<short-slug>.md` — no date prefix (the date lives in the frontmatter; the project README tracks shipped dates). The `phase-N` prefix establishes ordering when sorted alphabetically. Use `phase-Na`, `phase-Nb`, etc. for parallel sub-phases.

Both live under `~/workspace/second-brain/_meta/handoffs/` — standalone at the root, project-folder handoffs inside `<project-slug>/`.

## Wikilink target rules

Inside handoff bodies, follow the vault-wide slug-only wikilink convention from `~/workspace/second-brain/_meta/conventions.md`:

- Correct: `[[handoff-2026-05-27-multi-chat-coordination-skill]]`
- Wrong: `[[_meta/handoffs/handoff-2026-05-27-multi-chat-coordination-skill]]`

Obsidian resolves wikilinks by slug regardless of folder location, so file moves don't break links — but only if the wikilink is slug-only. The exception is when ambiguity exists (two files with the same slug in different folders) — then path-prefixed wikilinks are necessary.

## Tag rules

At minimum, every handoff tags:

- `handoff` (required, always)
- The project slug if part of a project folder (e.g., `intel-routing` for `intel-routing-rollout/` children, or `client-seo-onboarding` for `roadmap-client-seo-onboarding-automation/` children)
- Domain tags if applicable (`seo`, `automation-systems`, `content-systems`, etc.)
- Cross-cutting tags if applicable (`skill-build`, `keelworks-infrastructure`, `client-work`)

Tags drive Dataview queries elsewhere in the vault. Adding tags is cheap; under-tagging makes handoffs hard to find later.

## YAML quoting traps (avoid)

The tracker frontmatter's `last-change:` rule applies less often to individual handoffs, but the same trap exists for any field whose value contains a colon. Safe forms:

- Use single quotes when in doubt: `purpose: 'Build X. Includes Y: the thing.'`
- Plain strings work when the value has no colons or special chars: `purpose: Build X — see Y for context.`
- Never end a string value with a backslash (YAML continuation rules get weird).

## What the skill checks (AUDIT mode)

For every handoff cited in a tracker row, AUDIT confirms:

1. The file exists at the cited path.
2. The frontmatter parses as valid YAML.
3. The `type:` value is `handoff`.
4. The `status:` value matches the table the row lives in:
   - In-flight rows → `status: active` (handoff is currently being worked, not yet flipped to consumed)
   - Recently closed rows → `status: consumed`
   - Ready-to-spawn / Tier-2 / Tier-3 rows → `status: active`
5. For consumed handoffs: `consumed:` date is present, and either `actual-deliverable:` frontmatter field OR an "Actual deliverable" blockquote at the top of the body is present.
6. The `purpose:` field is non-empty.
7. The `tags:` list includes `handoff`.

Mismatches are surfaced in the AUDIT drift report as `STATUS DRIFT` or `FRONTMATTER ISSUES` per the drift-report-template.md format.
